## 1. 项目概述

### 1.1 项目简介

**守望者量化交易系统（warden-stock-quant）** 是一套基于 **A 股日线级别行情** 的**低频量化研究与交易平台**。它在 [`warden-stock-data`](../warden-stock-data)（行情数据中台）之上，沉淀「**策略 → 回测 → 分析 → 因子研究 → 仿真/实盘 → 仓位 → 风控 → 告警**」的完整量化闭环，并以**多用户 SaaS** 形态对外提供服务。

- **数据来源**：消费 `warden-stock-data` 的只读开放 API（`/open/v1/*`，HMAC 签名）获取日 K、复权因子、技术指标、证券列表、交易日历等；并支持其**全市场 K 线导出文件**的批量导入，构建本地高性能数据集（point-in-time，无未来函数）。
- **核心能力**：日线级别策略回测、绩效分析报告、量化因子研究、仿真/实盘交易、仓位管理、风险控制、日志告警监控。
- **服务对象**：量化研究者、个人/团队投资者；系统多租户隔离，可部署到线上供多用户使用。

### 1.2 与 `warden-stock-trending` / `warden-stock-data` 的关系

```
warden-stock-data（行情数据中台，Go + Python）
        │  /open/v1/*（只读 HMAC 行情 API）+ 全市场K线导出文件
        ▼
warden-stock-quant（本项目，Python 量化平台）  ◄── 灵感来源 warden-stock-trending（早期耦合系统）
        │  策略 / 回测 / 因子 / 仿真实盘 / 仓位 / 风控 / 告警
        ▼
   Web 控制台（React） + 多用户
```

- `warden-stock-data` 负责「**数据**」：只读、公共、可复用。本项目是其**只读消费方（Consumer）**，绝不向其写入。
- `warden-stock-trending` 是早期把「行情 + 策略 + 持仓 + 风控 + AI」耦合在一起的交易系统，是本项目的**灵感与需求来源**。本项目重构其量化部分，做到职责清晰、数据/策略/执行解耦。
- 本项目沿用守望者 **Scan · 侦查** 的能力定位：把全市场行情转化为可被策略、因子、组合复用的**可计算 alpha 底座**。

### 1.3 产品定位

| 维度 | 定位 |
|------|------|
| 频率 | **日线级别低频**（日频调仓 / 周频 / 月频），不做高频、不做 Tick 级 |
| 市场 | A 股（`market=CN`），预留多市场维度 |
| 模式 | 多用户 SaaS；研究 + 回测 + 仿真为主，实盘为可选自托管能力 |
| 角色 | 平台管理员（运营）+ 量化用户（研究/交易）|
| 数据 | 只读消费 `warden-stock-data`，本地构建可回放数据集 |

### 1.4 文档目标与范围

本 PRD 面向产品、研发、测试，定义**做什么（What）与为什么（Why）**，技术实现细节（How）见 `BACKEND.md` / `FRONTEND.md` / `QUANT_ENGINE.md` / `DATA_ACCESS.md`，接口契约以 `openapi.yaml` 为准。

**范围内（In Scope）**：M1–M11 模块（见 §5）。
**范围外（Out of Scope）**：

- 高频 / Tick 级策略、期货 / 期权 / 加密货币（仅预留扩展位）。
- 行情数据的采集与生产（属 `warden-stock-data` 职责）。
- 真实资金的全自动 A 股下单在「公网多租户」形态下不提供（合规受限，见 §3.4、§5 M8、§11.3）。

## 2. 名词术语表

| 术语 | 英文 / 缩写 | 说明 |
|------|------------|------|
| 数据集 | Dataset | 本地落地的、可按日回放的行情/指标快照（来自 data 服务的 API 或导出文件）|
| Point-in-Time | PIT | 任一历史交易日只能看到该日及之前的数据，杜绝未来函数 |
| 策略 | Strategy | 用户定义的交易逻辑（信号生成 + 下单规则），有多个版本 |
| 回测 | Backtest | 在历史数据上模拟运行策略，产出净值、交易、绩效 |
| 因子 | Factor | 对截面/时序股票打分的可计算指标（如动量、价值、技术因子）|
| IC / IR | Information Coefficient / Ratio | 因子值与未来收益的相关性 / 其稳定性 |
| 分层回测 | Quantile Backtest | 按因子值分组（如 5 档）观察各组收益，检验单调性 |
| 组合 | Portfolio | 一组目标持仓权重 + 资金，对应一个仿真/实盘账户 |
| 再平衡 | Rebalance | 按策略信号将组合调整到目标权重 |
| 仿真/纸面交易 | Paper Trading | 用真实行情但虚拟资金撮合，验证实盘前可靠性 |
| 实盘 | Live Trading | 经券商网关下真实订单（合规受限，自托管）|
| 风控 | Risk Control | 事前/事中/事后的仓位、暴露、回撤、止损等约束 |
| 网关 | Gateway | 交易执行的券商/模拟适配器（vnpy 风格）|
| 复权 | Adjust | `qfq`(前复权)/`hfq`(后复权)/不复权，回测必须口径一致 |
| 滑点/手续费 | Slippage / Commission | 撮合时对成交价/成本的模型化扣减 |
| 任务 | Job / Task | 异步执行单元（回测、因子计算、数据同步等）|
| 租户隔离 | Tenancy Isolation | 用户间数据严格隔离，互不可见 |

## 3. 用户、角色与权限模型

### 3.1 角色定义

| 角色 | 说明 | 典型操作 |
|------|------|---------|
| **平台管理员（Admin）** | 运营/维护者 | 用户管理、配额与套餐、数据源凭证管理（`warden-stock-data` 的 secretId/secretKey）、全局数据集同步、系统监控与告警渠道、审计日志 |
| **量化用户（User）** | 注册的普通租户 | 策略 CRUD、回测、查看报告、因子研究、组合与仿真交易、配置个人风控与告警；在自托管授权下可接入实盘 |
| **（可选）只读协作者** | 被共享的查看者 | 查看被授权共享的策略/报告（只读）｜后续迭代 |

### 3.2 权限模型（核心约束）

1. **多租户隔离铁律**：用户的策略、回测、因子、组合、订单、日志**严格按 `user_id` 隔离**，任何业务查询都强制注入当前用户作用域，越权访问返回 `403`。
2. **管理员边界**：管理员管理「平台级资源」（用户、配额、全局数据集、数据源凭证、系统告警），**默认不查看用户私有策略内容**（审计场景除外，且需记录审计日志）。
3. **数据源凭证集中托管**：`warden-stock-data` 的 `secretId/secretKey` 由平台统一持有与签名，用户**不直接接触**，仅通过本平台访问行情；避免密钥泄露与配额失控。
4. **实盘授权独立开关**：实盘交易能力默认关闭，需管理员对租户单独授权，且仅在自托管/单租户合规部署下可用（见 §3.4）。
5. **认证方式**：用户用 **JWT（OAuth2 Password Flow）** 登录；对程序化访问发放**个人 API Key（Bearer）**，可设 scope 与配额。

### 3.3 配额与套餐（多用户运营）

| 配额项 | 说明 |
|--------|------|
| 并发回测数 | 单用户同时运行的回测任务上限（队列排队）|
| 回测时间跨度 / 标的数 | 单次回测的历史区间与股票池规模上限 |
| 因子计算配额 | 单日因子计算次数 / 标的×日规模 |
| 存储配额 | 回测结果、报告、因子值的存储上限 |
| 数据 API 调用 | 经平台访问 `warden-stock-data` 的频次（平台侧再限流）|

配额按套餐（free/pro/...）配置，超额返回 `429`，由管理员设定。

### 3.4 合规边界（实盘交易）

> **重要**：真实 A 股程序化交易在国内有强合规门槛——需**专业投资者认证 + 券商三方系统采购 + 券商内网托管运行（禁止公网接入）**（来源：vnpy 社区 XTP/华鑫/东证接口说明）。因此：

- **公网多租户形态**：仅提供 **仿真/纸面交易** 与 **信号生成 + 人工执行/跟单提醒**，**不提供真实资金自动下单**。
- **自托管/单租户形态**：在用户具备资质并完成券商合规流程后，可接入 vnpy 真实券商网关（XTP/tora/OST/EMT 等）做实盘，平台提供执行/风控/监控能力。
- 所有交易相关页面均展示**投资风险免责声明**；本系统不构成投资建议。

## 5. 功能模块详细需求

> 每个模块给出：目标、功能点、输入输出、关键规则、验收标准。技术实现见 `BACKEND.md` / `QUANT_ENGINE.md`。

### M1 用户与多租户体系

**目标**：支撑多用户安全使用与运营管理。

**功能点**

- 用户注册 / 登录 / 登出 / 改密 / 找回（邮箱）；JWT（access + refresh）。
- 个人 API Key 管理：创建/吊销/轮换，设 scope（read/backtest/trade）与配额；密钥**仅创建时明文返回一次**。
- 角色与权限：`admin` / `user`；RBAC 中间件。
- 配额与套餐：管理员为用户配置套餐与各项配额（§3.3）。
- 审计日志：登录、密钥变更、实盘授权、关键操作落审计表。

**关键规则**

- 密码 bcrypt/argon2 哈希；JWT 失效与刷新；登录失败限速。
- 所有业务查询强制 `user_id` 作用域；越权 `403`。

**验收标准**

- 两个用户的策略/回测互不可见；API Key 越权被拒；配额超限 `429`。

---

### M2 行情数据接入与数据集模块

**目标**：把 `warden-stock-data` 的行情统一为本地、可按日 PIT 回放的高性能数据集，向回测/因子/实盘提供一致的数据访问层（DataFeed）。

**功能点**

1. **HMAC 数据客户端**：封装 `/open/v1/*`（indices/quotes/kline/intraday/indicators/search/securities/meta）的签名调用与重试退避（详见 `DATA_ACCESS.md`）。
2. **数据集导入**：
   - **API 增量同步**：按交易日历增量拉取日 K（含复权口径）、PIT 指标、证券列表、ST/停牌/退市/涨跌停标记。
   - **导出文件导入**：批量导入 data 服务的全市场 K 线导出文件（CSV/Parquet），首次全量最快路径。
3. **本地数据集存储**：日 K（OHLCV + 复权因子 + 状态位）、交易日历、证券元数据、指标快照；分区/索引优化按 `(code, date)` 与 `(date)` 两类访问。
4. **DataFeed 抽象**：向回测/因子/实盘提供统一接口：`get_bars(codes, from, to, period, adjust)`、`get_universe(date)`、`get_indicators(codes, date, types)`、`trading_calendar()`、`is_tradable(code, date)`（停牌/退市/未上市过滤）。
5. **复权与口径一致性**：回测默认前复权（`qfq`），口径在数据集与引擎间强一致。
6. **数据新鲜度与缺口检测**：暴露 `freshness`，自动探测缺口并补齐；标记 `stale`。

**输入输出**

- 输入：data 服务 API / 导出文件、同步任务参数。
- 输出：本地数据集、同步作业进度、数据质量报告。

**关键规则**

- **PIT 铁律**：任何 `date` 的回放只暴露该日及之前数据；指标用 data 服务的 PIT 快照或本地一致重算。
- 数据源凭证集中托管（§3.2），用户不接触。

**验收标准**

- 同一标的同一区间，API 同步与文件导入结果一致；回测在缺口/停牌日不产生伪成交；切换复权口径净值符合预期。

---

### M3 策略管理模块

**目标**：让用户以**安全、可复现、可版本化**的方式定义日线级别策略。

**功能点**

1. **策略 CRUD + 版本管理**：每次保存生成不可变版本（version），记录代码、参数 schema、依赖数据/因子；回测绑定具体版本以可复现。
2. **两类策略定义方式**：
   - **配置式（低代码）**：基于内置信号积木（指标交叉、因子打分、排名选股、定期再平衡）通过表单/JSON 配置，**无需写代码**，安全可控（M3 首选交付）。
   - **代码式（高级）**：用户编写 backtrader `Strategy` 子类（受限沙箱执行，白名单导入、禁用 IO/网络/文件、CPU/内存/超时限制）。
3. **参数 schema**：声明可调参数（类型、范围、默认值），供回测/寻优网格化。
4. **策略模板库**：内置经典策略（双均线、动量轮动、布林带、RSI、因子选股、行业中性等）作为起点。
5. **股票池（Universe）定义**：全市场 / 指数成分 / 自定义列表 / 因子筛选动态池。

**关键规则**

- 代码式策略**必须**在沙箱执行，禁止访问网络/文件系统/环境变量；只能经 DataFeed/Context API 取数。
- 策略只表达「信号与下单意图」，撮合/资金/风控由引擎统一处理。

**验收标准**

- 同版本策略 + 同数据 + 同参数 → 回测结果可完全复现（确定性）；恶意代码（读文件/联网）被沙箱拦截。

### M4 回测引擎模块

**目标**：在历史数据上高保真模拟日线策略，产出可分析的结果。

**功能点**

1. **引擎**：以 **backtrader** 为主引擎（事件驱动、自带指标与 analyzer），通过 `IBacktestEngine` 抽象封装，预留 vectorbt（参数寻优）/ zipline-reloaded（因子）切换（见 `QUANT_ENGINE.md`）。
2. **A 股交易规则建模**：
   - **T+1**：当日买入不可当日卖出。
   - **涨跌停**：触及涨停不可买入、跌停不可卖出（用 data 服务涨跌停价/状态）。
   - **停牌/退市/未上市**：不可交易，跳过或顺延。
   - **最小交易单位**：买入按 100 股（1 手）取整。
   - **复权口径**：默认前复权。
3. **成本模型**：佣金（双边，最低 5 元可配）、印花税（卖出 0.05% 可配）、过户费、滑点（固定/比例/成交量冲击）。
4. **资金与撮合**：初始资金、按 next-bar open/close 或 vwap 撮合、滑点后成交、现金/持仓账户更新。
5. **任务化运行**：回测以**异步任务（Celery）**执行，支持队列、并发配额、进度上报、取消、超时。
6. **参数寻优（可选）**：网格/随机搜索，批量回测 + 结果汇总（防过拟合提示）。
7. **结果落库**：净值序列、每笔交易、每日持仓、绩效指标、引擎日志。

**输入输出**

- 输入：策略版本 + 参数 + 股票池 + 区间 + 初始资金 + 成本/撮合配置。
- 输出：`backtest` 记录（状态/进度）、净值曲线、交易明细、持仓快照、绩效指标。

**验收标准**

- 已知策略基准用例结果在容差内可复现；T+1/涨跌停/停牌规则有单测覆盖；长区间×大股票池可在配额内完成且可取消。

### M5 绩效分析与报告模块

**目标**：把回测结果转化为专业、可读、可导出的分析报告。

**功能点**

1. **核心绩效指标**：累计/年化收益、波动率、夏普、索提诺、卡玛、最大回撤（及区间）、胜率、盈亏比、换手率、Alpha/Beta（对基准）、信息比率、月度收益表。
2. **曲线与可视化数据**：净值曲线（策略 vs 基准）、回撤曲线、收益分布、滚动夏普、持仓集中度、行业暴露随时间。
3. **交易与持仓明细**：逐笔交易（开平/盈亏/持有期）、每日持仓快照、归因（行业/个股贡献）。
4. **基准对比**：默认沪深 300 / 指定指数（取自 data 服务 indices）。
5. **报告导出**：HTML / PDF（含图表与指标表），可分享链接（只读、可设有效期）。
6. **多回测对比**：并排比较多个回测/参数组的关键指标。

**实现建议**：指标计算用 `quantstats` / `empyrical-reloaded`，报告模板化生成（见 `QUANT_ENGINE.md`）。

**验收标准**

- 指标与业界口径一致（夏普年化系数、回撤定义）；报告导出完整可读；对比视图数据正确。

### M6 量化因子研究模块

**目标**：提供单/多因子的定义、计算与有效性检验平台。

**功能点**

1. **因子定义**：
   - 内置因子库：动量（多周期 pct_change）、反转、波动率、量价（量比/换手）、技术因子（MA 乖离、MACD/RSI/BOLL 衍生）、规模/估值（如可得）。
   - 自定义因子：基于 DataFeed 的截面/时序表达式或受限代码。
2. **因子计算**：按交易日批量计算全市场/股票池因子值（异步任务），落 `factor_values`（PIT）。
3. **单因子检验**：
   - **IC/IR**：因子值与未来 N 日收益的秩相关（RankIC）、IC 均值/IR/胜率/衰减。
   - **分层回测**：按因子值分 N 档（默认 5/10），各档净值、单调性、多空组合（Top-Bottom）。
   - **换手率/覆盖度**：因子稳定性与可投资性。
4. **多因子合成**：标准化（去极值/中性化：市值/行业中性）、加权（等权/IC 加权/回归），生成合成因子。
5. **因子 → 策略**：合成因子可一键生成「因子选股策略」进入 M4 回测。

**实现建议**：分析用 `alphalens-reloaded` 风格指标 + pandas/numpy/scipy/statsmodels（见 `QUANT_ENGINE.md`）。

**验收标准**

- 因子值 PIT 无未来函数；IC/分层结果与手工核对一致；中性化后行业/市值暴露显著下降。

---

### M7 仓位与组合管理模块

**目标**：管理目标持仓与资金，连接策略信号与执行。

**功能点**

1. **组合（Portfolio）**：名称、初始资金、基准、关联策略、再平衡频率（日/周/月）、目标权重方案（等权/市值加权/因子加权/风险平价可选）。
2. **目标持仓与再平衡**：根据策略/因子信号生成目标权重，计算调仓指令（买卖股数），考虑 100 股取整、最小变动阈值、现金缓冲。
3. **持仓与资金跟踪**：实时/盘后持仓、可用资金、市值、浮动盈亏、累计收益（仿真账户）。
4. **组合分析**：组合净值、与基准对比、当前行业/个股暴露、集中度。
5. **与执行联动**：组合再平衡指令交给 M8 执行（仿真/实盘）。

**验收标准**

- 再平衡指令满足取整与阈值规则；持仓/资金账实一致；组合净值与成交一致可对账。

### M8 仿真/实盘交易模块

**目标**：把策略/组合信号转化为订单并执行，覆盖仿真（默认）与实盘（自托管合规）。

**功能点**

1. **执行引擎**：**vnpy 风格**的事件驱动执行内核 + **可插拔网关（Gateway）**：
   - **PaperGateway（默认）**：用真实行情（取自 data 服务）以纸面资金撮合，遵循 A 股规则与成本模型；线上多租户主用。
   - **券商网关（自托管）**：vnpy 的 XTP / tora(华鑫) / OST(东证) / EMT(东财) 等，仅在合规与授权下启用（§3.4）。
2. **运行模式**：
   - **盘后调度调仓**：交易日盘后用收盘信号生成次日订单（低频日频主用法）。
   - **信号推送/跟单提醒**：生成调仓建议并经告警渠道推送，用户人工执行（公网多租户合规友好）。
3. **订单生命周期**：创建→风控校验（M9）→提交→部分/全部成交→完成/拒绝/撤单；状态机 + 落库。
4. **账户管理**：仿真账户资金/持仓；实盘账户对接网关查询持仓/资金/成交回报。
5. **交易日历感知**：仅交易日运行；T+1 与涨跌停在执行层再次校验。

**关键规则**

- **实盘默认关闭**，需管理员对租户授权 + 自托管部署；公网形态禁用真实自动下单。
- 所有订单先过风控（M9），不通过则拒单并告警。

**验收标准**

- Paper 账户撮合与回测口径一致；订单状态机正确；风控拦截生效；实盘网关（自托管）能连通并回报。

---

### M9 风险控制模块

**目标**：统一的风控规则引擎，回测、仿真、实盘共用。

**功能点**

1. **事前风控（下单前）**：单票最大仓位比例、单行业/板块暴露上限、股票池白/黑名单、禁止买入 ST/退市整理/新股、单笔/单日下单金额上限、可用资金校验、涨跌停/停牌不可交易。
2. **事中风控（持仓中）**：组合最大回撤熔断（触发停止开仓/清仓）、单票止损/止盈、移动止损、波动率/集中度预警。
3. **事后风控（盘后/定期）**：风险敞口报表、违规记录、限额使用率。
4. **规则配置**：按用户/组合配置规则集（启用/阈值/动作：拒单/告警/强平），管理员可设平台级强制规则。
5. **风控事件**：每次触发落 `risk_events` 并触发 M10 告警。

**关键规则**

- 风控引擎与执行引擎解耦（独立可测）；回测中同样应用以贴近实盘。

**验收标准**

- 各规则有单测；触发动作正确（拒单/告警/强平）；回撤熔断在阈值处生效。

### M10 日志、告警与监控模块

**目标**：保障系统可观测、可追溯、可预警。

**功能点**

1. **结构化日志**：统一 JSON 日志（请求、任务、引擎、网关），含 `trace_id` / `user_id` / `job_id`。
2. **审计日志**：登录、密钥、实盘授权、下单、风控触发、配置变更，独立留存、可检索。
3. **任务监控**：回测/因子/同步/交易任务的状态、耗时、失败原因、重试；队列堆积监控。
4. **告警渠道**：邮件、Webhook、钉钉、飞书、Server酱等；用户可配置个人渠道，管理员配置平台渠道。
5. **告警规则**：任务失败、数据缺口/陈旧、风控触发、回撤超阈、实盘异常、配额耗尽。
6. **指标与健康检查**：`/healthz`、Prometheus 指标（请求/任务/队列/引擎耗时）。

**验收标准**

- 关键事件可在日志/审计检索到；告警按渠道送达且可去重/限频；健康检查与指标可用。

## 6. 非功能性需求（NFR）

| 类别 | 要求 |
|------|------|
| 性能 | 单次「5 年 × 300 标的」日频回测在配额内分钟级完成；接口 P95 < 500ms（不含异步任务）|
| 并发 | 回测/因子任务经队列削峰，单用户并发受配额限制 |

| 一致性 | PIT 无未来函数；回测确定性可复现；账实一致 |

## 7. 技术方案建议（详见技术文档）

> 本项目后端以 **Python** 为权威实现（量化生态 backtrader/vnpy/pandas/alphalens 均在 Python，且用户明确要求 Python 后台）。这是针对本仓库量化计算特性的**项目级技术决策**，在 `AGENTS.md` 通用技术表之上做的合理特化，理由见 `BACKEND.md` §1。

- **后端**：Python 3.11 + **FastAPI** + uvicorn/gunicorn；SQLAlchemy 2.0 + Alembic；Pydantic v2；**Celery + Redis** 任务队列；JWT/OAuth2 鉴权。
- **量化引擎**：**backtrader**（回测主引擎，抽象封装）；**vnpy**（仿真/实盘执行内核 + 网关）；pandas/numpy/scipy/statsmodels；`quantstats`/`empyrical-reloaded`（绩效）；`alphalens-reloaded` 风格因子分析；可选 vectorbt（寻优）/ zipline-reloaded（研究）。
- **数据接入**：httpx HMAC 客户端消费 `warden-stock-data` `/open/v1/*`；pandas/pyarrow 导入导出文件；本地数据集落 PostgreSQL（+可选 Parquet 列存）。
- **存储**：PostgreSQL（业务 + 数据集）+ Redis（缓存/队列/限流/锁）+ 文件/对象存储（报告/导出）。
- **前端**：React + Vite + shadcn/ui + Tailwind CSS；ky + TanStack Query；zustand；图表 ECharts / lightweight-charts；代码编辑器 Monaco。
- **部署**：Docker + docker-compose（postgres / redis / api / worker / beat / frontend）；PostgreSQL 数据 bind mount 到 `backend/deploy/pgdata`（入 `.gitignore`）；镜像名带项目名前缀 `warden-stock-quant-*`。
- **测试**：TDD 测试先行（pytest），引擎口径用例先行。

7. **实盘形态（已确认）**：公网多租户**仅提供仿真/纸面交易 + 信号推送**，真实资金自动下单**仅自托管单租户**合规部署可用（§3.4）。
8. **代码式策略（已确认）**：**首期只做配置式（低代码）**，代码式沙箱（M3 代码式 + 引擎沙箱）**延后到阶段五**实现；首期 API/数据库保留 `type=code` 字段位但不开放。
9. **大规模因子计算（已确认）**：**引入 Parquet + DuckDB 列存加速**层（PostgreSQL 为权威源，Parquet 为只读派生加速层，见 `DATA_ACCESS.md` §3.2、`QUANT_ENGINE.md` §5）。
10. **指数成分 / 行业分类 / 市值（已确认，待上游）**：`warden-stock-data` 暂未提供对应开放接口，后续会支持。当前**预留接口与字段、相关能力先留空/降级**：
    - 股票池 `type=index`（按指数成分选股）与因子**行业/市值中性化**先做接口占位，标记 `not_implemented`，待上游接口就绪再接入；
    - 临时降级：股票池可用「全市场 / 自定义列表 / 因子筛选」；行业维度可暂用 `securities.board`（板块，非细分行业）做粗粒度近似，市值中性化暂缺省关闭。
